You are working in my OGym codebase (React/Vite/TS + Wouter + React Query; Express/TS; Passport session auth; Postgres + Drizzle; schema in shared/schema.ts). We already have two cycle behaviors users can choose when creating a workout cycle: (1) Calendar-based and (2) Completion-based.

TASK:
Implement flexible “Rest/Reschedule/Train Anyway” behavior for BOTH:
A) Gym Mode cycles (trainer-assigned to members)
B) Personal Mode cycles (self-created)
while respecting the chosen cycle behavior (calendar-based vs completion-based).

NON-NEGOTIABLE:
- Must work for both trainer cycles (source="trainer") and personal cycles (source="self").
- Must not break existing workflows.
- Must be predictable and easy for users to understand.
- Do not silently change plans without an explicit user action.
- Keep data isolated (personal cycles never show in trainer/owner views; trainer cycles remain gym-scoped).

========================
FEATURE REQUIREMENTS
========================

1) Completion-Based Cycles (Sequence-based, flexible)
Definition:
- The “Next Workout” is the next uncompleted cycle day in sequence (not tied to date).
- The cycle advances ONLY when the user completes a workout day.

UI Behavior:
On Today/Workout screen for completion-based cycles, show:
- Primary: “Start Next Workout”
- Secondary: “Rest Today”
- Optional: “Pick Different Day”

Actions:
A) Rest Today:
- Logs a rest event for today (if we track rest), BUT does NOT advance the cycle.
- The next workout remains the same workout day.

B) Train on a planned rest day:
- If the next day in sequence is a Rest day OR the user is viewing a Rest day, show “Train Anyway”.
- “Train Anyway” should offer:
  1) “Do Next Workout (recommended)” -> starts next uncompleted workout day
  2) “Pick a Different Day” -> user selects a workout day from the cycle and completes it today

Completion:
- Completing a workout day marks that cycle day as completed for that date.
- The next workout is then the next uncompleted day in sequence.

Implementation Notes:
- Do NOT tie completion-based progression to calendar dates.
- Ensure both trainer and personal cycles use the exact same logic.

2) Calendar-Based Cycles (Date-based, structured)
Definition:
- The plan follows scheduled dates (e.g. Day 1 = Monday relative to startDate).
- The app shows “Today’s scheduled day” based on date.
- Flexibility is supported ONLY via explicit rescheduling actions.

UI Behavior:
On Today/Workout screen for calendar-based cycles, show:
- Primary: “Start Today’s Scheduled Workout”
- Secondary: “Reschedule”
- Optional: “Train Today (Extra)”

Actions:
A) User wants Rest on a scheduled workout day:
Button: “Reschedule”
Provide options:
  1) “Swap with next Rest day” (recommended):
     - Today becomes Rest
     - The scheduled workout is moved to the next Rest day within the current cycle window
  2) “Move to Tomorrow (shift schedule forward)”:
     - Move today’s workout to tomorrow and push subsequent scheduled days forward by 1
  3) “Skip (mark missed)” (optional, not recommended)

B) User wants to Train on a scheduled Rest day:
Button: “Train Today”
Provide options:
  1) “Train Extra (does not change schedule)”:
     - Logs an extra workout session but keeps calendar schedule unchanged
  2) “Swap with next workout day” (recommended):
     - Do the next scheduled workout today
     - Make that future day Rest or adjust schedule accordingly

Implementation Notes:
- Do NOT silently rewrite the calendar schedule without user choosing swap/move.
- Scheduled day mapping must remain correct after reschedule.
